# Background

Nanopublications are a novel approach to scholarly communication that represent the smallest possible units of publishable information. They are designed to be a formalized and machine-readable way to share knowledge, making it understandable and actionable for both humans and computers. Anything can be published through the decentralized nanopublications network, including: metadata about a scientific paper, comments, approvals, ammendments, verifiable proof of being part of an organization, and much more. The underlying network forms an open and trustworthy knowledge-graph with high semantic fidelity which, for example, can be used as a basis of facts for LLMs to process, reason or generate full publications specific to users or audiences. A nanopublication is an RDF document that has 3 required graphs:

- Assertion
- Provenance
- Publication Info

Find out more about nanopublications [here](https://nanopub.net/).

Overall, the Science Live app aims to: retain underlying noise-free semantic fidelity of nanopublications while providing a friendly frontend for a much wider user-base.

Science Live Platform is a web app that allows researchers to more easily view and create nanopublications on the nanopub decentralized network. A user can create an account, link it with their ORCID (via OIDC in Better Auth), and create signed nanopublications, which others can view and reference in their bibliographies for their own scientific work. Templates can be used or created, in order to create different types of nano nanopublications.

Science Live aims to open nanopublications to a much broader user-base, making nanopublications more intuitive, by layering a simpler UX on top of the complex underlying formal RDF-based format. This includes producing and querying RDF via natural language (potentially using AI models or standard algorithms) and a friendly user-interface for both creation and viewing, while preserving the benefits of underling formal RDF semantic fidelity such as machine readibility, validation, query, citability, versioning and knowledge graph traversal.

Verification of researchers and nanopublications is an important part of the system. Users will be able to verify users and nanopubs, and everything works in a decentralized way using signatures and hashing (Trusty URI). The platform includes organization and team invites and membership.

In future, Science Live may also feature a credits system which incentivises researchers to publish on the platform and improve it.

Many of these details are still to be determined.

# Architecture

There is a Frontend (`frontend` folder) and a Backend (`api` folder).

- **Frontend**
  - React 19
  - React Router
  - Tailwind v4
  - Shadcn/ui
  - Vite
  - Wrangler
- **Backend** (api)
  - Hono
  - Drizzle ORM
  - Better Auth
  - Postgres DB
  - Bun
  - Wrangler

Default deployment is on Cloudflare Workers.

Additionally there is a `zotero` folder which is the Science Live Zotero plugin, and a `homepage` folder which is a Gatsby static website serving as the product home page. Both of these re-use React components from the `frontend` workspace.

## Use Drizzle properly

- Don't create or modify files under `api/src/db/migrations` as these are generated by the command `npx drizzle-kit generate` after you make changes to the schema `*.ts` files under `api/src/db/schema`.
- To apply the generated migration to the database use `npx drizzle-kit migrate` but be very careful as it might be connected to a production or shared database. It is best to leave the `migrate` command to the user, just remind them to run it.
- Even though we are using better-auth, never use `npx @better-auth/cli migrate` nor `npx auth migrate` (v1.5+) for migrations, as we want to stick to drizzle for database management.

# Forms for creating nanopublications using Templates

Under `frontend/src/pages/np/create/components/templates/` you will find a number of user-friendly forms for creating popular types of nanopubs using templates (which are themselves nanopubs, listed in `frontend/src/pages/np/create/components/templates/registry-metadata.ts`).

The lightweight library `formedible` provides simplified form generation functionality with reduced boilerplate (using TanStack Form, Zod and shadcn/ui). Use this approach for complex forms and templates for creating Nanopublications, where possible. More information, including examples, are availble in the file [agent-docs/FORMEDIBLE.md](agent-docs/FORMEDIBLE.md). You can also follow some of the existing forms under `frontend/src/pages/np/create/components/templates/` as examples but feel free to improve on it.

The forms gather the required information defined in the template ("placeholders"), then pass that to the `generateNanopublication()` function which generates the "statements" (RDF triples) and the rest of the output nanopub, including signing it and generating a globally unique trustyHash to indentify it.

Before submitting the form data to generate the nanopub, sometimes it needs to be translated from the formedible form data format to the expected statement-based format, but only for cases such as:

- Advanced placeholders (object arrays mapping to `RepeatableStatement`).
- Advanced form components (API-retrieved search box that returns object that needs to be mapped to a string).
- Special data types (Convert dates to ISO date strings, possibly with time stripped off if only the date is important).

For example:

```ts
/* The `value` submitted from formedible:  */

// value = {
//   organization: "my-org",
//   datasource: "https://example.org/my-data",
//   topics: [
//     { uri: "https://example.org/topics/computing", label: "Computing" },
//     { uri: "https://example.org/topics/opensource", label: "Open Source" },
//   ],
//   disciplineSelection: {
//     uri: "https://example.org/disciplines/engineering",
//     label: "Engineering"
//   },
//   date: new Date();
// };
...

onSubmit: async ({ value }) => {
  const v = value as any;
  // Map topics to st4 (RepeatableStatement placeholder)
  v.st4 = v.topics?.map((k: { uri: string }) => ({
    topic: k.uri,
  }));
  v.discipline = v.disciplineSelection?.uri || "";
  if (v.date instanceof Date) {
    v.date = v.date.toISOString().split("T")[0];
  }

  // There is no need to delete `v.topics` etc, as extraneous values are simply
  // ignored by generateNanopublication() which is called in submit(), and
  // deletion would also remove the value from the form which the user may want
  // to go back and edit.
  await submit(v);
}
```

# Custom views for viewing nanopubs

Try to create and/or re-use common sub-components of each view in `frontend/src/pages/np/view/shared-components.tsx`, where it makes sense to do so.
